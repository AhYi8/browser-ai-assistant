# AI Agent 浏览器插件架构设计与实现方案 (Manifest V3)

## 一、 架构总览：三位一体的 Agent 引擎

为了在严格的 Manifest V3 CSP 限制下，实现一个能深度解析数据且能跨页面自动化操作的 AI Agent，本插件采用“层级一（DOM 操作）+ 层级二（后台调度）+ Sandbox（安全执行）”的复合架构。

### 核心模块与职责分工

- **大语言模型 (LLM / Agent):** 负责决策。不直接输出执行代码，而是输出标准化的 **JSON 格式的动作指令** (Action Payload)、**数据提取代码** (Extraction Code) 以及 **与用户沟通的对话**。
- **Content Script (层级一 - “手和眼”):** 注入到目标网页。负责执行不需要刷新的 DOM 操作（点击、滚动、输入），以及抓取网页原始 HTML 发给 Sandbox。
- **Background Service Worker (层级二 - “大脑”):** 插件后台。负责跨页面的状态流转、URL 跳转、新标签页管理，以及与 LLM 的核心网络通信。
- **Sandbox Iframe (沙盒 - “精算师 & 压缩机”):** 隐藏在插件内部的隔离页面。唯一允许执行 `eval()` 或 `new Function()` 的地方。它不仅负责安全执行 LLM 生成的 JS 代码解析 HTML，还负责在将 HTML 发给 LLM 前进行**体积压缩**。

## 二、 核心通信协议 (JSON Contract)

为了支持“先沟通，再执行”的 Battle 机制，LLM 必须且只能输出符合以下结构的 JSON（Schema）。这是整个系统运转的齿轮。

```
{
  "thought": "用户要求抓取列表，但我发现页面底部有分页器，我需要先问清楚抓取几页（Battle阶段）。",
  "reply_to_user": "我已经看到了商品列表。请问您只需要提取当前页，还是需要我自动点击『下一页』帮您抓取前 5 页的数据？",
  "plan_ready": false, // 标记是否已经和用户达成一致并准备好执行
  "extraction": {
    "need_extract": false,
    "code": "" 
  },
  "action": {
    "type": "none",       // 可选值: none, dom_action (层级一), navigate (层级二)
    "command": "",        // 当 type 为 dom_action 时生效，如 click, scroll, input
    "selector": "",
    "url": ""             // 当 type 为 navigate 时生效
  }
}
```

## 三、 逐步实现蓝图 (给 Codex 的任务拆解)

### 阶段零：需求对齐 (Battle) 与自动化编排 (核心体验革新)

- **功能目标:** 在盲目开始自动化操作之前，AI 必须先结合“用户模糊的意图”和“当前真实的网页结构”进行分析，主动向用户提问以明确边界，最后生成可执行的自动化工作流。
- **具体步骤:**
    1. **意图与上下文捕获:** 用户在 UI 面板输入初步需求（例如：“把这里的订单导出来”）。Content Script 立刻抓取当前页面结构发给 LLM。
    2. **多轮澄清 (Battle 机制):** * LLM 收到 HTML 后，发现订单有时间筛选项。
        - LLM 返回 JSON，其中 `plan_ready: false`，并在 `reply_to_user` 字段中提问：“我发现页面上有日期筛选，您是要导出所有历史订单，还是只导出本月的订单？需要我先帮您选择本月吗？”
    3. **共识与编排:** 用户回答“只要本月，前3页即可”。LLM 明确需求后，向用户展示一个即将执行的操作步骤（SOP），询问是否确认开始。
    4. **进入执行态:** 用户点击确认后，LLM 开始输出 `plan_ready: true`，并在后续的回调中依次输出 `action` 和 `extraction` 指令，正式驱动阶段一、二的代码执行。

### 阶段一：搭建 Sandbox 数据提取管线 (突破 CSP 提取数据)

- **功能目标:** 让 LLM 能够动态写 JS 来提取网页数据。
- **具体步骤:**
    1. **创建沙盒文件:** 在项目根目录创建 `sandbox.html` 和 `sandbox.js`。
    2. **配置 Manifest:** 在 `manifest.json` 中添加 `"sandbox": { "pages": ["sandbox.html"] }`。
    3. **建立通信隧道:**
        - 在 Background 或 SidePanel 中，通过隐藏的 `<iframe>` 加载 `sandbox.html`。
        - 使用 `window.postMessage` 将 `{html_string, llm_code}` 发给沙盒。
        - `sandbox.js` 中使用 `new Function('htmlString', llm_code)` 执行代码（注入 DOMParser 解析 htmlString），然后将结果 `postMessage` 传回。
- **⚠️ 避坑指南:**
    - _坑点:_ Sandbox 无法直接访问 Content Script 所在网页的真实 DOM。
    - _解法:_ Content Script 必须先使用 `document.documentElement.outerHTML` 拿到纯文本源码，发给 Sandbox，Sandbox 内部使用 `new DOMParser().parseFromString(html, 'text/html')` 重建一个虚拟 DOM 树供 LLM 代码查询。
    - _安全注意:_ 务必检查 `event.origin` 以防止跨站脚本攻击。

### 阶段二：实现层级一 (Content Script DOM 自动化)

- **功能目标:** 在不刷新页面的情况下，执行点击、滚动等操作。
- **具体步骤:**
    1. 在 Content Script 中实现一个 `ActionExecutor` 类。
    2. 接收 Background 传来的 `action` JSON。
    3. 如果是 `click`: 执行 `document.querySelector(selector).click()`。
    4. 如果是 `scroll`: 执行 `window.scrollTo(...)`。
    5. 执行完毕后，等待一定时间（解决 DOM 渲染延迟），重新抓取 HTML 发给 Sandbox。
- **⚠️ 避坑指南:**
    - _坑点 (Timing Issue):_ 触发 click 或 scroll 后，React/Vue 渲染新数据需要时间，如果立刻抓取 HTML 会抓到老数据。
    - _解法:_ 引入 `MutationObserver` 监听 DOM 树变化，或者使用智能的 `setTimeout` 轮询检测目标元素是否出现后再进行下一步。
    - _坑点 (伪造点击失效):_ 某些网站的按钮绑定了原生的 `mousedown/mouseup` 而不是 `click`。
    - _解法:_ 封装一个强大的 `simulateClick` 函数，依次派发 `pointerdown`, `mousedown`, `mouseup`, `click` 事件。

### 阶段三：实现层级二 (Background 跨页面状态管理)

- **功能目标:** 处理需要刷新或跳转的场景，保证 Agent 的任务“不断联”。
- **具体步骤:**
    1. 在 Background Script (Service Worker) 中建立一个状态机 (State Machine)，记录当前 Task 的状态和已抓取的数据（使用 `chrome.storage.session`）。
    2. 当解析到 `action.type === 'navigate'` 时。
    3. Background 调用 `chrome.tabs.update({url: action.url})`。
    4. 监听 `chrome.tabs.onUpdated`，当该 Tab 的 status 变为 `complete` 时，重新向该 Tab 注入/唤醒 Content Script，并下发“继续提取”的指令。
- **⚠️ 避坑指南:**
    - _坑点 (Service Worker 休眠):_ Manifest V3 的 Background 脚本在闲置 30 秒后会被强制挂起 (Suspending)。如果 LLM 请求时间过长，后台可能会死掉，导致回调失败。
    - _解法:_ * 方法 A (推荐): 使用 `chrome.alarms` 定时唤醒，或在通信期间保持 Native Messaging 的端口连接以延长生命周期。
        - 方法 B: 将核心的 LLM 网络请求移到 SidePanel 或独立的 Offscreen Document 中执行，它们不会像 Service Worker 那样被轻易挂起。
    - _坑点 (SPA 与多页的误判):_ 有些网站 URL 变了但实际上是用 History API 实现的无刷新跳转。
    - _解法:_ Background 监听 `chrome.webNavigation.onHistoryStateUpdated`，如果触发了这个，就不要完全重载逻辑，而是通知现有的 Content Script 页面已更新。

### 阶段四：组装流水线 (Main Loop)

1. **交互起点:** 用户在 UI 输入初步意图。
2. **需求 Battle 循环:** UI 请求当前页 HTML -> LLM 分析 -> 生成 `reply_to_user` 询问用户 -> 用户回复 -> 循环直至 LLM 输出 `plan_ready: true`。
3. **获取上下文 (执行态):** 执行第一步预定动作前，确保拿到最新状态的 HTML 并送入 Sandbox。
4. **执行提纯:** 插件解析 JSON 中的 `extraction.code`，将数据送入 Sandbox，将结果存入全局 Storage。
5. **执行动作:** * 若是 `dom_action`，交给 Content Script，完成后回到步骤 3。
    - 若是 `Maps`，交给 Background 跳转，页面 Load 完成后回到步骤 3。
6. **终止:** LLM 的 JSON 返回 `action.type: 'none'`，任务结束，向用户汇报并展示收集到的所有数据。

## 四、 阶段五：进阶与健壮性优化 (生产级必做)

在基础架构跑通后，为了应对复杂的真实互联网环境，必须实现以下四重保障机制。

### Token 爆炸优化 (DOM 清洗与压缩)

- **痛点:** 现代网页的 HTML 动辄几 MB，直接发给 LLM 会瞬间超 Token 限制或导致 API 响应极慢。
- **解法:** 在每次发送页面上下文给 LLM 之前，在 Sandbox 中执行一段预处理代码：
    - **清洗 DOM:** 剔除所有 `<script>`, `<style>`, `<svg>`, `<img>`, 以及无用的 `class` 和 `style` 属性。
    - **降维转换:** 可选将清理后的 HTML 转换为精简版 Markdown，或者生成 Accessibility Tree (AOM 辅助功能树)。LLM 理解 Markdown 结构比理解复杂 HTML 要好得多，并且 Token 消耗减少 90%。

### 错误恢复与自我修正 (Self-Correction Loop)

- **痛点:** LLM 生成的 `selector` 不存在，或者 Sandbox 执行 `code` 报错，导致整个流程崩溃。
- **解法:** Content Script / Sandbox 执行失败时，绝对不能停止！必须生成一份错误报告回传。
    - 构建一个新的 Message 给 LLM：`"上一条指令执行失败，错误信息: TypeError: Cannot read properties of null (reading 'click') at selector '.next-btn'。请重新分析当前页面并提供新的策略。"`
    - 引入**重试次数上限** (如 3 次)，超过则中止执行并向用户抛出异常说明。

### 穿透现代框架 (Shadow DOM & Iframe 处理)

- **痛点:** 越来越多的网站（如 Twitter, Youtube）大量使用 Web Components (Shadow DOM)。`document.querySelector` 默认是**无法**查找 Shadow DOM 内部元素的。嵌套的 iframe 也无法跨域抓取。
- **解法:**
    - **Shadow DOM 穿透:** 指导 Codex 在 Sandbox 中准备一个 `deepQuerySelector` 工具函数，利用递归或 `element.shadowRoot` 来查找深层元素，并允许 LLM 的 JS 代码调用此函数。
    - **Iframe 注入:** 在 `manifest.json` 的 `content_scripts` 必须加上 `"all_frames": true`。并设计一种机制，让 Content Script 能判断目标元素是在哪个 Frame 中，从而定向发送指令。

### 人性化延迟与反爬虫对抗

- **痛点:** Action 执行过快，一秒内点击 5 次，会被 Cloudflare 等 WAF 判定为高频爬虫直接拦截。
- **解法:**
    - 在 Content Script 的 `ActionExecutor` 中硬编码**随机延迟机制** (Random Delay)。比如翻页后强制休眠 800ms ~ 2000ms。
    - 模拟真人阅读行为：在执行 `click` 之前，先触发该元素的 `mouseover` 和 `mousemove` 甚至 `focus` 事件。
