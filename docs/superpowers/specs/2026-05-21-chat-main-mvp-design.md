# 聊天主界面 MVP 设计

## 背景

渠道管理和提取规则已经完成 MVP 落地，主界面需要从当前占位聊天面板演进为可实际对话、可保存历史、可按页面上下文辅助回答的 Chrome Side Panel 聊天工作台。

现有代码已经具备以下基础：

- `App` 中有聊天区、设置区和历史会话占位。
- `ChatSession`、`ChatMessage`、`chatSessions` 存储表已经存在。
- 渠道和模型已拆分为 `ModelProvider` 与 `ProviderModel`。
- 内容脚本已经支持按 URL 规则匹配 CSS/XPath 并提取页面可见文本。
- background 已支持页面上下文提取的内容脚本缺失时注入后重试。

## 目标

- 实现主聊天界面的响应式 MVP。
- 宽面板显示左侧历史对话模块，右侧显示聊天区。
- 窄面板隐藏左侧历史模块，在顶部模型行提供历史记录按钮并通过弹窗查看历史。
- 支持历史对话加载、重命名、归档、删除确认和文件夹分组。
- 支持文件夹折叠；已归档默认折叠并固定在历史区域底部。
- 支持“提取文本 / 提取所有”上下文模式。
- 每次用户发送消息时，将当前会话的全部消息提交给 AI。
- 支持 AI 消息气泡、用户消息气泡和 AI 思考过程折叠显示。
- 保持现有 Claude light 主题方向，符合 Side Panel 紧凑工具型体验。

## 非目标

- 不实现上下文消息数量配置；后续可增加只带最近 N 轮。
- 不抓取外链 CSS、JS 或其他跨域资源。
- 不实现拖拽排序文件夹或会话。
- 不实现历史搜索、标签、批量管理。
- 不实现多端同步冲突处理。
- 不把已归档作为普通文件夹管理。

## 组件设计

### App

`App` 继续负责顶层布局、设置页与聊天页切换、宽窄面板入口。它不直接承载聊天业务逻辑，避免顶层组件继续膨胀。

### SessionList

宽面板左侧历史对话模块，负责：

- 文件夹展示。
- 文件夹折叠状态。
- 会话选择。
- 会话重命名。
- 会话归档。
- 会话删除确认。
- 新建会话入口。

文件夹标题左侧不显示展开或收起箭头符号。普通文件夹可以折叠，默认文件夹不可删除但可以折叠。

### SessionHistoryDialog

窄面板历史记录弹窗。它复用 `SessionList` 的核心数据和操作逻辑，只改变承载方式，避免宽窄面板交互分叉。

### ChatPanel

右侧聊天主区域，组合以下模块：

- 当前模型行。
- 消息列表。
- 聊天输入区。
- 上下文状态和刷新入口。

窄面板时，当前模型行右侧显示历史记录按钮。

### MessageList

只负责消息渲染：

- AI 消息在左侧。
- 用户消息在右侧。
- AI 使用机器人头像。
- 用户使用默认用户头像。
- AI 思考过程存在时，在 AI 最终回答气泡上方显示折叠模块。

### ChatComposer

输入区负责：

- 文本输入。
- 发送按钮。
- 流式响应 Switch。
- 提取模式 Switch。
- 查看上下文入口。
- 匹配规则名展示。
- 刷新上下文按钮。

提取模式 Switch 是单个开关：

- 关闭状态显示“提取文本”。
- 开启状态显示“提取所有”。

## 数据模型

### ChatSession

在现有结构上最小扩展：

- `id`
- `title`
- `folderId?: string`
- `archived: boolean`
- `sortOrder`
- `createdAt`
- `updatedAt`
- `messages`

`folderId` 为空或为默认文件夹 ID 时，归入默认文件夹。`archived` 为 `true` 时，会话显示在“已归档”分组中。

### ChatFolder

新增聊天文件夹：

- `id`
- `name`
- `sortOrder`
- `createdAt`
- `updatedAt`

删除文件夹时不删除会话，而是把该文件夹下的会话移回默认文件夹。

### ChatMessage

在现有结构上最小扩展：

- `thinking?: string`
- `contextMode: "text" | "all"`
- `matchedRuleId?: string`

`contextPrompt` 继续保存本次实际发送给模型的页面上下文内容。它可能是可见文本，也可能是 HTML 截断结果。

## 历史会话交互

### 加载与新建

- 点击会话加载该会话的全部消息。
- 新对话入口创建空会话，默认放入默认文件夹。
- 发送第一条消息后可以根据用户消息生成默认标题；标题生成模型能力可后续增强，MVP 可先使用用户消息前若干字符。

### 重命名

重命名只修改会话标题，不改变会话 `updatedAt` 排序，避免用户整理标题时打乱历史顺序。

### 归档

归档将 `archived` 设置为 `true`。归档后的会话不再显示在普通文件夹中，而是显示在“已归档”固定分组内。

“已归档”默认折叠，并固定在历史区域底部。即使普通会话过多导致历史列表滚动，“已归档”仍贴在底部区域。

### 删除确认

删除采用两段状态：

1. 用户点击“删”，该会话进入待确认态，按钮文案变为“确认”。
2. 用户再次点击“确认”后才真正删除。

切换会话、点击其他历史操作或超时后，应退出待确认态。

## 响应式布局

### 宽面板

建议在 `min-width: 720px` 左右进入宽面板布局：

- 左侧为历史对话模块。
- 右侧为聊天对话区。
- 历史栏宽度保持紧凑，避免挤压聊天主区域。

### 窄面板

窄面板移除左侧历史栏：

- 顶部模型行显示“当前模型”和模型下拉。
- 模型行右侧显示历史记录按钮。
- 点击历史记录按钮打开历史弹窗。
- 规则名过长时截断，悬停或点击可查看完整名称。

### 聊天气泡

- 气泡最大宽度按容器比例限制。
- AI 消息左对齐，用户消息右对齐。
- 按钮点击区域保持约 `40px`，降低 Side Panel 误触。

## 页面上下文提取

扩展现有 `pageContext.extract` 消息，增加 `extractMode: "text" | "all"`。

### 提取文本

关闭提取模式开关时使用 `extractMode: "text"`：

- 如果 URL 命中提取规则，则按 CSS/XPath 提取匹配节点的可见文本。
- 如果没有命中规则，则提取全局可见文本。
- 如果命中规则但 CSS/XPath 未提取到内容，则退回全局可见文本。

### 提取所有

开启提取模式开关时使用 `extractMode: "all"`：

- 如果 URL 命中提取规则，则按 CSS/XPath 提取匹配节点的完整内容。
- 元素节点优先提取 `outerHTML`。
- 文本节点提取文本内容。
- 如果没有命中规则，则提取完整 `document.documentElement.outerHTML`。
- 如果命中规则但 CSS/XPath 未提取到内容，则退回完整 HTML。

### 通用规则

- 两种模式都必须执行 `maxLength` 截断。
- 两种模式都返回 `truncated` 状态。
- 两种模式都返回 `matchedRuleId`，用于 UI 显示匹配规则名。
- 窄面板中，如果匹配到规则则显示规则名；没有匹配则显示“全局文本”。
- 如果 URL 命中规则但选择器未提取到内容，UI 应提示“规则命中但无内容，已回退”，避免用户误以为当前上下文完全来自该规则。
- 不主动抓取外链 CSS、JS 或其他跨域资源。
- 不执行页面脚本。

## 聊天请求

新增 `chat.send` runtime 消息，由 Side Panel 发起，background 执行模型请求。

发送前校验：

- 用户输入非空。
- 当前模型存在且启用。
- 当前模型所属渠道存在且启用。
- 渠道 API Key 和 Endpoint 满足已有模型校验规则。

每次发送消息时，请求必须包含当前会话的全部消息：

- 当前模型的 `systemPrompt`。
- 当前页面上下文 `contextPrompt`。
- 当前会话中已有的全部用户和 AI 消息。
- 本次新输入的用户消息。

后续如果增加“上下文消息数量”配置，应集中在请求消息构造函数中处理，例如 `buildChatRequestMessages`，不应把截断策略散落到 UI 组件中。

## 流式响应

UI 保留流式响应 Switch，`ChatMessage.streamMode` 记录本次发送使用的流式状态。

实现优先级：

1. MVP 必须实现非流式真实模型请求。
2. 流式开关状态应传入请求。
3. 如果普通 runtime message port 无法稳定承载长流式请求，本轮可以不实现逐 token 刷新，但不得移除 UI 开关和数据字段。
4. 后续真实流式响应优先使用可控的端口连接或事件通道，避免长任务挂在普通 `sendMessage` 上。

## AI 思考过程

`ChatMessage.thinking` 只在检测到思考内容时存在。

MVP 支持 `<think>...</think>` 形式：

- 从模型回复中解析 `<think>` 内容。
- 将思考内容保存到 `thinking`。
- 从最终回答中剥离 `<think>` 块。
- UI 中仅当 `thinking` 存在时显示“AI 思考过程”折叠模块。

如果某些模型返回非标准 reasoning 字段，只有在当前适配器能稳定获取时才映射到 `thinking`；否则不在 MVP 中强行实现。

## 错误处理

- 未配置可用模型：禁用发送按钮，提示先配置渠道和模型。
- 当前页面无法注入内容脚本：上下文区域显示中文错误，不阻塞用户继续纯聊天。
- 页面上下文提取失败：保留输入能力，显示错误和“刷新”按钮。
- 模型请求失败：保留用户消息，不保存失败 AI 消息，显示错误卡片和“重试”按钮。
- 流式中断：AI 占位消息标记失败，不保存为正式完成消息。
- 完整 HTML 被截断：上下文区域显示“内容已截断”，请求仍可继续。
- 删除确认态只对当前点击的会话生效。

## 测试验收

### 单元测试

- `extractMode="text"`：命中规则、未命中规则、规则命中但无内容、截断状态。
- `extractMode="all"`：命中规则提取 `outerHTML`、未命中规则提取完整 HTML、规则命中但无内容、截断状态。
- 会话存储：创建会话、保存消息、重命名、归档、删除。
- 文件夹存储：创建文件夹、重命名文件夹、删除文件夹后会话回默认文件夹。
- 聊天请求构造：请求包含当前会话全部消息、本次用户消息、页面上下文和系统提示。
- 思考解析：`<think>...</think>` 被剥离并保存到 `thinking`。

### 组件测试

- 宽面板显示历史栏。
- 窄面板显示历史按钮和历史弹窗。
- 删除从“删”进入“确认”，再次点击才删除。
- 文件夹折叠后隐藏会话。
- 已归档默认折叠并显示在底部。
- AI 思考过程仅在存在 `thinking` 时显示。
- 提取模式开关关闭显示“提取文本”，开启显示“提取所有”。

### Background 与构建验证

- `chat.send` 成功返回模型回复。
- `chat.send` 请求失败返回中文错误。
- 无可用模型或渠道时返回中文错误。
- `npm test` 通过。
- `npm run build` 通过。
- 构建后检查 `dist/content/index.js` 不包含静态 `import` 或动态 `import(`。

## 风险

- 完整 HTML 上下文可能很大，必须截断并明确提示。
- 某些页面 HTML 可能包含用户敏感信息，UI 应清楚表达当前提取模式。
- 流式响应在 MV3 runtime message 中存在生命周期风险，MVP 不应依赖普通 `sendMessage` 长时间保持连接。
- 不同模型的思考过程格式不统一，MVP 只稳定支持 `<think>...</think>`。
- 当前会话全部消息都提交给 AI，长会话可能超出模型上下文；后续需要增加上下文数量或 token 预算配置。

## 验收标准

- 宽面板和窄面板布局符合视觉模板。
- 历史会话可以加载、重命名、归档、删除。
- 文件夹可以折叠，已归档默认折叠并固定在底部。
- 删除会话必须经过“删 → 确认”。
- 发送消息时包含当前会话全部消息。
- “提取文本 / 提取所有”逻辑符合本设计。
- AI 回复显示在左侧，用户消息显示在右侧。
- 只有检测到思考内容时才显示 AI 思考过程折叠模块。
- 相关单元测试、组件测试和构建验证通过。
