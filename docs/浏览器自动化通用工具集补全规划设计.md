# 浏览器自动化通用工具集补全规划设计

## 1. 背景与目标

本项目的核心定位是浏览器自动化助手，而不是只服务 Web 逆向。现有工具集已经覆盖基础页面操作、Network 采集、JS/Source Map 分析、运行时只读读取、受控增强请求确认、请求重放沙箱和完全访问模式，但工具组织仍偏“按实现来源分组”，缺少面向用户和任务的统一分类。

本规划的目标是在继续补全通用浏览器自动化工具前，先规范化现有工具集，并建立可扩展的工具分类体系。后续用户可以按运行要求、能力分类和风险级别快速筛选、批量启用或禁用工具，模型也可以基于分类更清晰地理解每个工具的使用边界。

## 3. 工具分类体系设计

### 3.1 设计原则

- 分类只描述工具运行要求、核心能力和总体风险，不替代现有 allow-list、运行态校验和参数校验。
- 分类字段必须是结构化枚举，禁止用自由文本驱动权限判断。
- 分类用于筛选、批量启用、UI 展示、模型提示词和测试断言；真正执行时仍由 background 二次校验。
- 新增工具必须先归类，再注册；不能只给 `groupId` 和 `requiredCapabilities`。
- 分类应足够粗，不把数据敏感度、输出附件、脱敏策略等实现细节都变成筛选维度；这些细节继续由工具描述、附件 kind、执行器和测试约束表达。

### 3.2 三项核心分类

#### 3.2.1 运行要求 `runtime`

用于回答“这个工具是否必须在 `chrome.debugger` 模式下使用”。

| 值 | 含义 | 示例 |
| --- | --- | --- |
| `local` | 不依赖网页或 debugger，只读本机/扩展环境信息 | `system.current_time` |
| `external_web` | 请求公开互联网服务 | `web_search.tavily` |
| `browser_control` | 必须开启浏览器控制并 attach 受控页 | `browser.take_snapshot` |
| `controlled_enhanced` | 必须处于受控增强模式 | `replay.send_request` |
| `full_access` | 必须处于完全访问模式 | `full_access.execute_script` |

#### 3.2.2 能力分类 `capabilities`

用于回答“这个工具主要解决哪类任务”。一个工具可以属于多个能力分类。

| 值 | 含义 |
| --- | --- |
| `observe_page` | 观察页面结构、状态或现场 |
| `operate_page` | 对页面产生交互或导航 |
| `analyze_site` | 分析页面、请求、源码或运行时 |
| `confirm_boundary` | 请求用户确认权限边界 |
| `deliver_result` | 生成或聚合用户可见结果 |
| `search_public_web` | 搜索公开网页资料 |
| `system_context` | 获取系统或会话辅助信息 |

#### 3.2.3 风险级别 `risk`

用于快速识别是否适合默认批量启用。

| 值 | 含义 | 示例 |
| --- | --- | --- |
| `low` | 只读、低副作用，通常适合默认启用 | `browser.take_snapshot` |
| `medium` | 会操作页面或读取较多页面现场，需要用户理解边界 | `browser.click` |
| `high` | 会发起请求、扩大读取范围或需要逐次确认 | `replay.send_request` |
| `critical` | 完全访问、任意脚本、原始凭据或原文读取 | `full_access.execute_script` |

#### 3.3.3 禁用规则

- 非 `chrome.debugger` 模式下，除 `system.current_time` 和 `web_search.tavily` 外，所有上述工具都必须在 UI 中置灰，不允许批量启用。
- background 也必须二次校验运行态，不能因为前端误显示启用就放行。
- 模型提示词中只能看到当前真实可用的工具，不能通过“手动勾选”绕过 debugger 依赖。

#### 3.3.4 落地拦截链路

后续实现工具分类时，必须按以下四层同时拦截，避免只在 UI 层禁用造成绕过：

1. 工具设置 UI：非 `chrome.debugger` 模式下，`runtime` 为 `browser_control`、`controlled_enhanced`、`full_access` 的工具必须置灰，并显示“需开启浏览器控制”。
2. 批量启用入口：全选、按分类启用、预设启用、导入配置恢复时，都必须跳过当前运行态不可用的工具。
3. 模型请求构造：发送请求前根据真实运行态过滤 `enabledToolIds`，不可把不可用工具 schema 和浏览器自动化系统提示词发给模型。
4. background 执行器：即使收到伪造工具调用，也必须基于当前 `BrowserControlManager` 连接状态、自动化模式和工具附加要求返回固定中文错误。

### 3.5 非核心分类信息

以下信息不再作为第一版筛选维度，避免维度过细：

- 是否会生成附件。
- 是否参与导出。
- 是否需要展示前重新脱敏。
- 是否读取 Network body、源码片段或运行时摘要。
- 是否包含一次性授权 grant。

这些信息继续放在工具描述、附件 kind、执行器返回值和测试约束中。UI 如需展示，可以从工具说明和运行时状态派生，不进入批量筛选分类。

### 3.6 分类结构建议

第一阶段已在 `ModelToolRegistryEntry` 上新增结构化字段，例如：

```ts
toolClassification: {
  runtime: "browser_control",
  capabilities: ["observe_page", "analyze_site"],
  risk: "medium",
}
```

现有 `requiredCapabilities` 继续保留，用于模型工具暴露过滤；`toolClassification` 用于 UI 筛选、批量选择、运行态过滤、提示词和审计展示。两者职责不应混用。

## 4. 分类化后的工具组织方案

### 4.1 推荐 UI 分层

工具选择界面建议从单一“浏览器自动化”分组升级为三层结构，但每层都保持粗粒度：

1. 工具来源：系统内置、公开网页搜索、浏览器自动化。
2. 能力域：观察页面、操作页面、分析现场、请求确认、交付结果。
3. 快速筛选：运行要求和风险级别。

用户可以通过能力域批量启用，也可以用风险级别快速禁用高危工具。例如：

- 只启用“观察页面 + 低风险”。
- 启用“基础操作”，但禁用高风险和最高风险工具。
- 启用“分析现场”，但不启用完全访问。
- 启用“完全访问”时仍必须由运行态开关控制，工具列表勾选不能自动进入完全访问模式。

### 4.2 批量启用策略

建议内置以下预设：

| 预设 | 包含能力 | 排除能力 |
| --- | --- | --- |
| `只读观察` | 快照、页面状态、Network 元数据、脱敏详情、Console 只读 | 页面操作、导航、请求重放、完全访问 |
| `基础操作` | 快照、点击、填写、按键、等待、滚动 | 请求重放、完全访问、任意脚本 |
| `页面巡检` | 快照、截图、Console、Network、页面状态 | 表单提交类高风险操作、完全访问 |
| `接口分析` | Network、JS、Source Map、Runtime 只读、请求重放草案 | 完全访问、任意脚本 |
| `受控增强分析` | 接口分析 + 边界确认 + 无凭据重放 | 完全访问 |
| `完全访问` | `full_access.*` | 仍由单独运行态控制，不能仅靠预设开启 |

预设只是批量修改用户启用工具列表，不替代运行态授权。

### 4.3 模型提示词策略

工具分类也应进入浏览器自动化系统提示词，用简洁规则告诉模型：

- 优先使用只读观察工具确认现场，再执行页面操作。
- 页面操作后要验证结果，例如重新快照、等待文本或检查 Network。
- 遇到运行要求为 `controlled_enhanced` 或 `full_access` 的工具时，必须遵守运行态边界。
- 对风险级别为 `high` 或 `critical` 的工具，不能把模型推断当作用户确认。
- 交付结论时要区分“工具证据”“模型推断”和“未验证假设”。

## 5. 五类能力缺口与补全规划

### 5.1 观察页面

#### 已有能力

- `browser.take_snapshot` 读取可访问树。
- `browser.list_pages` 列出受控页面。
- `network.*` 观察请求。
- `runtime.*` 读取受限运行时摘要。
- 完全访问模式可读取原始存储和 Network 原文。

#### 原始缺口与当前状态

- 原缺少截图观察，模型无法主动获取页面视觉状态；第一版已通过 `browser.screenshot` 和 `browser-screenshot` 附件落地。
- 原缺少 Console、页面异常、资源加载错误和 CSP 错误；第一版已通过 `browser.get_console_messages` 和后台 Console recorder 落地。
- 原缺少页面状态摘要，例如 URL、title、readyState、焦点元素、滚动位置、viewport；第一版已通过 `browser.get_page_state` 落地。
- 原缺少元素级 DOM 细节读取，例如属性、可见性、尺寸、computed style、disabled 原因；第一版已通过 `browser.inspect_element` 和 `browser.find_elements` 落地。
- iframe 和 shadow DOM 的可见边界仍不够完整，第一版保持保守读取和明确失败提示，后续如扩展必须继续避免任意 selector、XPath 或模型自定义脚本入口。

#### 建议新增工具

| 工具 | 作用 | 分类建议 |
| --- | --- | --- |
| `browser.get_page_state` | 返回当前 URL、title、readyState、viewport、滚动位置、焦点摘要、受控页面身份 | `browser_control` / `observe_page` / `low` |
| `browser.screenshot` | 截取当前视口或指定 UID 元素图片，供视觉模型分析或用户查看 | `browser_control` / `observe_page` / `medium` |
| `browser.get_console_messages` | 读取 Console 日志、JS 异常、资源错误摘要 | `browser_control` / `observe_page, analyze_site` / `medium` |
| `browser.inspect_element` | 按 UID 读取元素 DOM 属性、可见性、尺寸、可交互性原因 | `browser_control` / `observe_page, analyze_site` / `low` |
| `browser.find_elements` | 按文本、role、label、placeholder 或有限 CSS 查询候选元素，返回 UID 候选 | `browser_control` / `observe_page` / `low` |

#### 安全限制

- 截图必须限制尺寸、类型和附件数量；当前只允许 PNG，单张最大 5MB。
- `browser.screenshot` 的图片不得写入工具正文，只能通过 `browser-screenshot` 结构化附件返回；后续追问和导出只保留截图元数据，不注入 base64。
- 元素截图只接受 `take_snapshot` 返回的 UID，不接受 CSS、XPath、坐标猜测或模型自定义脚本。
- Console 日志要脱敏，避免泄露 token、Cookie、Authorization、URL 敏感参数。
- `find_elements` 的 CSS 查询必须限制长度和选择器复杂度，不允许任意脚本。
- `inspect_element` 不得返回完整 DOM 子树或超大 HTML。

### 5.2 操作页面

#### 已有能力

- 点击、填写、按键、等待文本。
- 导航、新建页面、切换页面、关闭页面。

#### 原始缺口与当前状态

- 原缺少滚动，导致长页面、虚拟列表、懒加载任务不稳定；第一版已通过 `browser.scroll` 落地。
- 原缺少 hover、双击、右键、拖拽，复杂后台和菜单操作受限；第一版已通过 `browser.hover`、`browser.double_click`、`browser.context_click` 和 `browser.drag` 落地。
- 文件上传和下载执行能力仍保持后续预留，不默认开放；第一版只落地文件上传、下载和本地文件访问边界确认。
- 原缺少等待页面状态的工具，例如等待 URL 变化、加载完成、元素可见/不可见、Network 空闲；第一版已通过 `browser.wait_for_state` 落地。
- 原缺少操作风险分级，普通点击也可能触发删除、付款、发布等副作用；第一版已通过工具分类风险级别、页面副作用边界检测和提示词约束落地。

#### 建议新增工具

| 工具 | 作用 | 分类建议 |
| --- | --- | --- |
| `browser.scroll` | 页面或指定 UID 元素滚动，支持方向、距离、顶部/底部 | `browser_control` / `operate_page` / `medium` |
| `browser.hover` | 悬停 UID 元素，用于展开菜单或提示 | `browser_control` / `operate_page` / `medium` |
| `browser.double_click` | 双击 UID 元素 | `browser_control` / `operate_page` / `medium` |
| `browser.context_click` | 右键 UID 元素，仅打开上下文菜单，不自动选择危险项 | `browser_control` / `operate_page` / `medium` |
| `browser.drag` | 从源 UID 拖到目标 UID 或坐标偏移 | `browser_control` / `operate_page` / `high` |
| `browser.upload_file` | 向 `<input type="file">` 选择用户确认的本地文件 | `browser_control` / `operate_page, confirm_boundary` / `high` |
| `browser.wait_for_state` | 等待 URL、readyState、元素可见性、Network 空闲等状态 | `browser_control` / `observe_page, operate_page` / `low` |

#### 安全限制

- `upload_file` 必须由用户选择或确认文件，模型不能凭空读取本地路径。
- `drag`、`context_click`、`double_click` 默认应标为中高风险，必要时进入受控增强确认。
- 操作后应支持 `includeSnapshot` 或建议调用验证工具。
- 对可能触发表单提交、删除、付款、发布的操作，第一版通过 `boundary.request_user_choice` 和页面副作用边界检测承接确认，不新增并行确认协议。

### 5.3 分析现场

#### 已有能力

- Network 请求列表、详情、等待、清空、对比。
- 参数候选识别。
- JS 资源检索和上下文提取。
- Source Map 映射和原始源码提取。
- 运行时只读摘要。
- 请求重放沙箱。
- 完全访问模式下原文读取和任意脚本。
- `browser.analyze_interaction_blocker` 第一版，用于按 UID 分析元素点击、填写或查看失败的常见阻塞原因。
- `browser.analyze_form` 第一版，用于分析页面或指定 UID 所属表单的字段状态、校验摘要、错误文案和提交按钮状态。
- `browser.get_performance_summary` 第一版，用于读取导航时序、资源耗时分组、慢资源和长任务摘要。

#### 原始缺口与当前状态

- 原 Console 与页面错误未进入分析链；第一版已通过 `browser.get_console_messages` 和 `browser.collect_diagnostics` 进入现场诊断链路。
- 性能和加载诊断已落地第一版，但后续仍可补充 Web Vitals、内存和更细的渲染瓶颈摘要。
- 表单失败诊断已落地第一版，但后续仍可补充错误文案与字段的关联关系、跨 Shadow DOM 表单和自定义组件表单。
- 元素不可交互原因分析已落地第一版，但后续仍可补充复杂层叠上下文、动画中状态和 Shadow DOM 遮挡判断。
- 原缺少跨工具证据聚合，工具结果分散在多条记录中；第一版已通过 `automation-report` 基于真实工具记录聚合步骤、时间线、失败摘要和证据。

#### 建议新增工具

| 工具 | 作用 | 分类建议 |
| --- | --- | --- |
| `browser.analyze_interaction_blocker` | 分析 UID 元素不可点击、不可填写或不可见原因 | `browser_control` / `analyze_site` / `low` |
| `browser.analyze_form` | 分析当前表单字段、必填项、错误文案、提交按钮状态 | `browser_control` / `analyze_site` / `medium` |
| `browser.get_performance_summary` | 返回导航时序、资源耗时摘要、长任务或加载状态 | `browser_control` / `analyze_site` / `low` |
| `browser.collect_diagnostics` | 聚合页面状态、Console、Network 错误和快照摘要 | `browser_control` / `observe_page, analyze_site, deliver_result` / `medium` |

#### 安全限制

- 性能和诊断摘要不得包含敏感 Header、Cookie 或完整响应体。
- 表单诊断只能读取字段状态和错误文案，不自动提交。
- 跨工具聚合必须保留每项证据的脱敏和截断标记，不能在聚合时恢复原文。

### 5.4 请求确认

#### 已有能力

- `boundary.request_user_choice`。
- 受控增强模式。
- 一次性边界授权。
- 完全访问模式和撤销。

#### 原始缺口与当前状态

- 原缺少针对页面副作用操作的确认，例如删除、发布、付款、发送消息；第一版已复用 `boundary.request_user_choice`，由边界检测和提示词要求模型先请求用户确认。

- 原缺少跨站点跳转和第三方授权页确认；第一版已复用 `boundary.request_user_choice`，由边界检测和提示词要求模型在离开当前站点或继续第三方授权流程前先请求用户确认。
- 原缺少“本次操作是否允许继续”的通用确认工具形态；第一版继续复用 `boundary.request_user_choice` 的动态问题、选项和一次性授权模型，不新增并行确认工具协议。

#### 建议新增机制

| 机制 | 作用 | 分类建议 |
| --- | --- | --- |
| `boundary.confirm_page_effect` | 对可能产生真实业务副作用的页面操作请求用户确认 | `controlled_enhanced` / `confirm_boundary` / `high` |
| `boundary.confirm_file_access` | 对上传、下载、读取文件元信息进行确认 | `controlled_enhanced` / `confirm_boundary` / `high` |
| `boundary.confirm_cross_site_navigation` | 对跳转到非当前 origin 或第三方授权页确认 | `controlled_enhanced` / `confirm_boundary, operate_page` / `high` |

这些机制可以先复用 `boundary.request_user_choice` 的 UI 与一次性 grant 模型，不急于新增完全独立的工具协议；第一版页面副作用确认、文件上传下载边界确认、跨站点跳转和第三方授权页确认已按该方式落地。

## 6. 分阶段实施计划

### 阶段 A：工具分类规范化

#### 目标

为现有所有注册工具补充结构化分类，并让 UI 可按分类筛选。

#### 当前落地状态

- 已扩展 `ModelToolRegistryEntry.toolClassification`，并为当前注册表内所有工具补齐 `runtime`、`capabilities`、`risk`。
- 已提供固定分类枚举和 `filterModelToolsByClassification`，用于工具列表筛选和批量操作。
- Side Panel 发送模型请求前已按用户启用列表、`toolClassification.runtime`、浏览器控制开关和自动化模式过滤工具；开启浏览器控制不再自动追加整组浏览器工具。
- 全局聊天偏好工具设置已支持按能力、运行要求和风险筛选，并可批量启用或禁用筛选结果；批量启用会跳过当前运行态不可用工具。
- 浏览器自动化系统提示词已按观察页面、操作页面、分析现场、请求确认、交付结果和风险级别约束模型调度顺序，要求先观察再操作、操作后验证，高风险或最高风险工具结果不得替代用户确认。
- 输入区当前聊天工具菜单已改为用户可勾选浏览器工具，但非当前运行态可用工具仍置灰。
- background 执行期仍保留现有 `shouldExposeTool` 和各工具执行器二次校验，防止伪造调用绕过 UI。

#### 范围

- 扩展 `ModelToolRegistryEntry` 类型。
- 为现有工具补齐 `toolClassification`。
- 增加分类枚举和校验测试。
- 工具设置 UI 增加分类筛选和预设批量启用。
- 非 `chrome.debugger` 模式下，所有 `runtime !== "local" && runtime !== "external_web"` 的工具必须置灰，批量启用也不能选中。
- 请求构造前必须按真实运行态过滤工具列表，避免把 debugger 工具 schema 或系统提示词发给模型。
- 保留现有 `requiredCapabilities` 过滤逻辑。

#### 验收标准

- 每个注册工具都有完整分类。
- 分类枚举非法时测试失败。
- 用户可以按“观察页面、操作页面、分析现场、请求确认、交付结果”筛选工具。
- 用户可以按“本地工具、公开网页搜索、浏览器控制、受控增强、完全访问”和风险级别筛选工具。
- 非 `chrome.debugger` 模式下，浏览器控制类工具无法通过单选、全选、预设或导入配置启用。
- background 对非 debugger 运行态下的伪造浏览器工具调用返回固定中文错误。

### 阶段 B：观察页面工具补全

#### 目标

补足浏览器自动化最常见的观察源，让模型先看清现场。

#### 当前落地状态

- 已新增 `browser.get_page_state`，用于读取当前受控页 URL、标题、`readyState`、viewport、滚动位置和焦点元素摘要。
- `browser.get_page_state` 不接受任何参数，只通过固定 `Runtime.evaluate` 模板读取只读页面状态，并对 URL 敏感 query、焦点文本等输出做脱敏与截断。
- 未开启浏览器控制时，`browser.get_page_state` 与其他 `browser.*` 工具一样置灰，并由 background 返回固定中文工具错误。
- 已新增 `browser.get_console_messages`，用于读取 background 通过 debugger 事件已采集的 Console 日志、JS 异常和资源错误摘要。
- `browser.get_console_messages` 不接受任何参数，不执行模型自定义脚本；输出只包含脱敏、截断后的日志正文和位置摘要。
- Console 缓存随浏览器控制关闭、页面切换、导航、刷新和 tab 关闭清理，避免旧页面现场污染新任务。
- 已新增 `browser.inspect_element`，用于按 `take_snapshot` 返回的 UID 读取元素属性、AX 摘要、布局、有限样式和可交互状态。
- `browser.inspect_element` 只接受 UID，不接受选择器或自定义脚本；输出不包含完整 DOM 子树或 `outerHTML`，并对文本和 URL 做脱敏截断。
- 已新增 `browser.find_elements`，用于在最近一次 `take_snapshot` 的 UID 候选中按文本、role、label、placeholder 或简单 CSS 查找元素。
- `browser.find_elements` 不扫描完整 DOM，不生成临时 selector；返回的 UID 可继续用于 `inspect_element`、`click` 或 `fill`。
- 已新增 `browser.screenshot`，用于截取当前视口或最近一次 `take_snapshot` UID 对应元素的 PNG 图片。
- `browser.screenshot` 必须在已 attach 的 `chrome.debugger` 会话中通过 `Page.captureScreenshot` 执行；工具正文只返回元数据摘要，图片通过 `browser-screenshot` 附件返回，后续追问和导出不注入 base64。
- `browser.screenshot` 当前只允许 PNG，单张最大 5MB；元素截图只接受 UID 并基于 `DOM.getBoxModel` 计算 clip，不接受选择器、XPath、坐标猜测或自定义脚本。

#### 范围

- `browser.get_page_state`（已落地）
- `browser.get_console_messages`（已落地）
- `browser.inspect_element`（已落地）
- `browser.find_elements`（已落地）
- `browser.screenshot`（已落地）

#### 验收标准

- 未开启浏览器控制时工具不可见且 background 拒绝伪造调用。
- 截图结果大小、类型、附件数量受限。
- Console 和页面状态输出脱敏。
- 元素检查不返回完整 DOM 或超大 HTML。
- 元素查找只返回最近快照中的 UID 候选，不返回完整 DOM 或临时 selector。
- 相关 vitest、`npm run typecheck`、`npm run build:extension` 通过。

### 阶段 C：操作页面工具补全

#### 目标

提升复杂页面操作能力，但继续保持明确边界。

#### 范围

- `browser.scroll`（已落地）
- `browser.hover`（已落地）
- `browser.double_click`（已落地）
- `browser.context_click`（已落地）
- `browser.drag`（已落地第一版）
- `browser.wait_for_state`（已落地第一版）
- 文件上传先只做设计或 behind flag，不默认开放。

#### 当前落地状态

- 已新增 `browser.scroll`，用于滚动当前视口或最近一次 `take_snapshot` UID 对应的可滚动元素。
- `browser.scroll` 必须在已 attach 的 `chrome.debugger` 会话中执行，默认滚动当前视口；提供 UID 时只通过快照映射解析元素，不接受 CSS、XPath、坐标猜测或模型自定义脚本。
- `browser.scroll` 的 `direction` 只允许 `up`、`down`、`left`、`right`、`top`、`bottom`；`amount` 必须是 1 到 5000 的整数，默认 800。
- `browser.scroll` 成功后可通过 `includeSnapshot` 追加最新页面快照，用于验证长页面、虚拟列表或懒加载内容变化；失败时不得自动追加快照。
- 已新增 `browser.hover`，用于把鼠标移动到最近一次 `take_snapshot` UID 对应元素中心，触发悬停菜单、提示层或 hover 状态。
- `browser.hover` 只接受 UID 和可选 `includeSnapshot`，不接受 CSS、XPath、坐标猜测或模型自定义脚本；执行时通过 `DOM.getBoxModel` 计算中心点并发送真实 `mouseMoved` 事件。
- `browser.hover` 成功后可通过 `includeSnapshot` 追加最新页面快照，用于验证菜单或提示层是否展开；失败时不得自动追加快照。
- 已新增 `browser.double_click`，用于双击最近一次 `take_snapshot` UID 对应元素，触发双击编辑、打开条目等交互。
- `browser.double_click` 只接受 UID 和可选 `includeSnapshot`，不接受 CSS、XPath、坐标猜测或模型自定义脚本；执行时通过 `DOM.getBoxModel` 计算中心点并发送真实双击鼠标事件序列。
- `browser.double_click` 成功后可通过 `includeSnapshot` 追加最新页面快照，用于验证编辑态或打开结果；失败时不得自动追加快照。
- 已新增 `browser.context_click`，用于右键最近一次 `take_snapshot` UID 对应元素，打开页面上下文菜单或自定义右键菜单。
- `browser.context_click` 只接受 UID 和可选 `includeSnapshot`，不接受 CSS、XPath、坐标猜测或模型自定义脚本；执行时通过 `DOM.getBoxModel` 计算中心点并发送真实右键鼠标事件序列。
- `browser.context_click` 只负责打开上下文菜单，不自动选择菜单项，也不得把“菜单已打开”扩展为“已执行菜单动作”；如需确认菜单内容，应通过 `includeSnapshot` 或后续观察工具验证。
- `browser.context_click` 成功后可通过 `includeSnapshot` 追加最新页面快照，用于验证菜单是否出现；失败时不得自动追加快照。
- 已新增 `browser.drag` 第一版，用于从最近一次 `take_snapshot` 的 `sourceUid` 拖拽到 `targetUid`，或按有限 `deltaX`、`deltaY` 相对偏移拖拽。
- `browser.drag` 被标记为高风险操作；第一版不接受 CSS、XPath、绝对坐标、坐标猜测或模型自定义脚本，也不提供任意 JS fallback。
- `browser.drag` 的 `deltaX`、`deltaY` 必须是 -2000 到 2000 的整数，且不能和 `targetUid` 同时提供；执行时通过 `DOM.getBoxModel` 计算元素中心点，并发送真实鼠标拖拽事件序列。
- `browser.drag` 成功后可通过 `includeSnapshot` 追加最新页面快照，用于验证拖拽结果；失败时不得自动追加快照。
- 已新增 `browser.wait_for_state` 第一版，用于等待 URL 片段、`document.readyState`、最近一次 `take_snapshot` UID 对应元素可见或隐藏，以及 Network 空闲。
- `browser.wait_for_state` 的 `state` 只允许 `url_contains`、`ready_state`、`element_visible`、`element_hidden`、`network_idle`；等待 URL 或 readyState 时必须提供非空 `value`，等待元素显隐时必须提供 UID，等待 Network 空闲不需要 `value` 或 `uid`。
- `browser.wait_for_state` 不接受 CSS、XPath、坐标猜测或模型自定义脚本；元素状态等待通过快照 UID 解析元素，并在固定 `Runtime.callFunctionOn` 模板中轮询自身布局状态。
- `browser.wait_for_state` 成功后可通过 `includeSnapshot` 追加最新页面快照，用于验证等待结果；失败时不得自动追加快照。
- Network 空闲等待已接入 `browser.wait_for_state` 第一版，基于 Network recorder 的进行中请求状态判断空闲窗口。

#### 验收标准

- 长页面和懒加载页面可以通过滚动稳定观察。
- hover 菜单、双击编辑、右键菜单等常见交互可完成。
- URL、readyState、元素显隐状态和 Network 空闲可以通过 `wait_for_state` 稳定等待。
- 拖拽操作已按高风险分类落地第一版；右键操作已按中风险收口为“仅打开菜单”。
- 操作后 UID 失效、导航、弹窗和 detach 处理保持现有规则。

### 阶段 D：现场分析工具补全

#### 目标

把浏览器自动化从“能操作”推进到“能排查为什么失败”。

#### 范围

- `browser.analyze_interaction_blocker`（已落地第一版）
- `browser.analyze_form`（已落地第一版）
- `browser.get_performance_summary`（已落地第一版）
- `browser.collect_diagnostics`（已落地第一版）

#### 当前落地状态

- 已新增 `browser.analyze_interaction_blocker` 第一版，用于按最近一次 `take_snapshot` 返回的 UID 分析元素不可点击、不可填写或不可查看的常见原因。
- `browser.analyze_interaction_blocker` 被归类为 `browser_control / analyze_site / low`，必须在已 attach 的 `chrome.debugger` 会话中启用；非 debugger 运行态下 UI 置灰且 background 拒绝伪造调用。
- `browser.analyze_interaction_blocker` 只接受 `uid` 和可选 `expectedAction`；`expectedAction` 只允许 `click`、`fill`、`view`，默认 `click`。
- `browser.analyze_interaction_blocker` 不接受 CSS、XPath、坐标、选择器猜测或模型自定义脚本；执行时只通过快照 UID 解析节点，并使用固定 `Runtime.callFunctionOn` 模板读取只读诊断字段。
- 第一版诊断范围包括元素可见性、禁用态、可编辑性、是否仍连接文档、是否在视口、`pointer-events`、中心点遮挡元素、禁用 fieldset 和表单非法字段数量。
- 输出必须结构化展示证据、阻塞原因和建议动作，并对文本、遮挡元素摘要等内容做脱敏与截断；工具只负责诊断，不修改 DOM、不触发表单、不自动重试操作。
- 已新增 `browser.analyze_form` 第一版，用于分析当前页面或最近一次 `take_snapshot` UID 所属表单的字段数量、必填项、非法字段、禁用字段、错误文案和提交按钮状态。
- `browser.analyze_form` 被归类为 `browser_control / analyze_site / medium`，必须在已 attach 的 `chrome.debugger` 会话中启用；非 debugger 运行态下 UI 置灰且 background 拒绝伪造调用。
- `browser.analyze_form` 只接受可选 `uid` 和可选 `includeFieldDetails`；`uid` 必须来自最近一次 `take_snapshot`，`includeFieldDetails` 必须是布尔值。
- `browser.analyze_form` 不接受 CSS、XPath、选择器、坐标或模型自定义脚本；未传 UID 时通过固定 `Runtime.evaluate` 模板分析页面表单，传 UID 时通过快照 UID 解析节点后分析该节点所属表单。
- `browser.analyze_form` 不返回用户输入原文值，字段详情只返回标签、类型、名称、必填、禁用、只读、非法和 `hasValue` 状态；错误文案、按钮文本、表单 action 等输出必须脱敏与截断。
- 已新增 `browser.get_performance_summary` 第一版，用于读取当前受控页面的 Navigation Timing、资源耗时分组、慢资源列表和 Long Task 摘要。
- `browser.get_performance_summary` 被归类为 `browser_control / analyze_site / low`，必须在已 attach 的 `chrome.debugger` 会话中启用；非 debugger 运行态下 UI 置灰且 background 拒绝伪造调用。
- `browser.get_performance_summary` 不接受任何参数，不接受模型自定义脚本；执行时通过固定 `Runtime.evaluate` 模板读取 `performance.getEntriesByType("navigation" | "resource" | "longtask")`。
- `browser.get_performance_summary` 只返回脱敏后的性能元数据，例如耗时、传输大小、资源类型和脱敏资源 URL；不得读取或输出 Header、Cookie、响应体、完整资源内容或敏感 query 原文。
- 已新增 `browser.collect_diagnostics` 第一版，用于一次性聚合当前受控页面的页面状态、Console、性能摘要和最近 Network 错误/慢请求摘要。
- `browser.collect_diagnostics` 被归类为 `browser_control / observe_page, analyze_site, deliver_result / medium`，必须在已 attach 的 `chrome.debugger` 会话中启用；非 debugger 运行态下 UI 置灰且 background 拒绝伪造调用。
- `browser.collect_diagnostics` 不接受任何参数；页面状态和性能摘要复用固定读取模板，Console 复用后台采集摘要，Network 只调用脱敏后的请求元数据列表。
- `browser.collect_diagnostics` 不读取响应体、Header、Cookie、请求体原文或完整资源内容；任一单项读取失败时只输出中文失败摘要，不让聚合诊断成为敏感原文恢复通道。

#### 验收标准

- 表单按钮 disabled、字段错误、元素不可点击等场景能给出结构化原因；`browser.analyze_interaction_blocker` 第一版已覆盖元素禁用、遮挡、不可见、不可编辑和表单非法字段摘要，`browser.analyze_form` 第一版已覆盖表单字段、按钮和错误文案摘要。
- 页面错误、Network 错误、性能摘要可聚合；`browser.get_performance_summary` 第一版已覆盖导航时序、慢资源和长任务摘要，`browser.collect_diagnostics` 第一版已覆盖页面状态、Console、性能和 Network 错误/慢请求聚合摘要。
- 聚合诊断不泄露敏感原文，不读取 Network 响应体、Header、Cookie、请求体原文或完整资源内容。

### 阶段 E：交付结果与任务报告

#### 目标

让用户能复盘浏览器自动化任务，而不是只看到零散工具调用。

#### 范围

- 新增 `automation.report` 附件 kind（已落地第一版，实际附件 kind 为 `automation-report`）。
- 设计自动化时间线结构（已落地第一版，随 `automation-report` 归一化、展示、导出和后续追问上下文一起流转）。
- 设计按任务类型组织的结构化报告（已落地第一版，随 `automation-report.reportType` 在 UI、导出和后续追问上下文中展示）。
- 最终回答中区分事实证据、模型推断和未验证假设（已落地第一版，通过工具循环最终总结指令约束）。
- 导出和后续追问支持报告摘要。

#### 验收标准

- 一次复杂任务完成后能看到清晰步骤、证据和结果。
- 第一版已在浏览器自动化工具循环结束后，基于真实 `toolCallRecords` 自动生成 `automation-report` 附件，包含任务类型、目标、结论、步骤证据、自动化时间线、失败摘要和完全访问标记。
- `automation-report` 已接入通用 `toolAttachments` 体系，支持历史归一化、消息展示、导出和后续追问上下文注入。
- 最终总结指令已要求模型把工具结果作为事实证据，把基于证据的判断标为模型推断，把未被工具或用户确认的信息标为未验证假设。
- 失败时能看到卡点、原因和可恢复动作；第一版通过失败步骤数、失败工具和统一可恢复动作摘要表达。
- 报告导出前重新脱敏；第一版会对目标、结论和步骤证据做敏感字段脱敏与截断。
- 完全访问结果是否原文保留必须有明确标记；第一版报告通过 `fullAccessIncluded` 标识本轮是否包含完全访问工具结果。

## 7. 推荐优先级

推荐顺序为：

1. 工具分类规范化。
2. 页面状态、截图、Console 三个观察工具。
3. 滚动、hover、等待状态三个操作工具。
4. 表单诊断和不可交互原因分析。
5. 自动化任务报告。

理由是：分类体系先落地后，后续新增工具都能自然进入筛选、批量启用、提示词和测试体系；观察工具优先可以提升所有任务类型的稳定性；操作工具再补能避免模型在看不清现场时盲目操作。

## 8. 风险与边界

- 分类筛选不能替代执行期权限校验。即使 UI 显示禁用，background 仍必须拒绝伪造工具调用。
- “完全访问”不能通过批量启用预设自动开启，只能由用户显式切换运行态。
- 截图、Console、DOM 属性和页面状态都可能包含敏感信息，必须默认脱敏和截断。
- 文件上传、下载、拖拽、右键菜单、表单提交类能力必须谨慎进入受控增强或逐次确认。
- 自动化报告不能成为绕过脱敏的通道，历史脏数据展示前仍必须重新脱敏。

## 9. 文档与测试要求

- 修改工具分类、工具注册或工具分组时，必须同步更新本文档和 `AGENTS.md`。
- 新增工具必须补充注册表测试、执行器参数校验测试、运行态暴露条件测试和必要 UI 测试。
- 涉及浏览器真实行为、截图、拖拽、文件上传、下载或扩展加载路径时，应追加 Playwright 扩展冒烟验证。
- 纯文档修改可不运行自动化测试，但必须至少检查 Markdown 内容和 Git diff。

## 10. 与现有规划的关系

`docs/浏览器自动化控制总体规划设计.md` 仍作为浏览器控制地基和阶段历史文档；本文档作为下一阶段“通用工具集补全与工具分类化”的详细规划。

`docs/Network工具化与Web逆向自动化规划.md` 继续作为 Network、JS、Source Map、Runtime、Replay 和 Full Access 的专题文档。后续新增通用工具时，不应把项目主线重新收窄到 Web 逆向；Network 能力应作为“分析现场”能力域的一部分参与统一分类体系。
