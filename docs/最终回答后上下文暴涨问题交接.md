# 最终回答后上下文暴涨问题交接

> 交接目标：让新的对话在没有历史上下文的前提下，可以继续分析并实现“最终回答结束后上下文暴涨”的修复。

## 背景

当前项目是 Chrome 扩展 `browser-ai-assistant`。近期已完成一轮大功能：工具循环内上下文压缩、运行中上下文估算、Network/JS 详情池、会话级工具附件仓库、持续引导对话等。

用户遇到的新问题是：工具循环最终回答结束后，底部圆形 Token 进度条显示的“实际请求上下文”突然暴涨，例如从压缩后的几十 k 上升到 `133k/100k`。用户质疑：既然已经有 Network 详情池、JS 详情池、附件仓库和上下文压缩，为什么最终回答结束后上下文不是只增加最终回答 response 的 token，而是大幅增加。

结论：这不是累计 token 的问题，也不是完整 Network/JS body/source 一定被重复注入；根因更接近“最终回答落库后，下一次真实请求上下文重新展开了最终回答消息引用的工具附件摘要”，并且存在压缩边界被最终回答附件引用绕过的风险。

## 已确认的代码链路

1. 底部 Token 圆环显示的是“下一次真实请求上下文”，不是上一轮 response token。
   - 入口：`src/side-panel/components/ChatComposer.tsx`
   - `currentContextTokens` 在非发送中状态下会调用 `buildChatRequestMessages(...)` 构造下一次请求消息，再调用 `estimateModelRequestContextTokens(...)` 估算。
   - 因此最终回答结束后，圆环会重新扫描当前会话历史、最终回答正文、工具附件摘要、Prompt 快照、图片等，而不是只看最终回答 token。

2. 下一次请求会展开 assistant 历史消息上的工具附件摘要。
   - 入口：`src/shared/chat/buildChatRequestMessages.ts`
   - `buildChatRequestMessages` 会对最新 `context_summary` 之后的历史消息执行 `expandAssistantContextAttachments(...)`。
   - assistant 消息上的 `toolAttachmentIds` 会通过 `toolAttachmentsById` 解析为附件，再调用 `formatToolAttachmentForPromptSummary(...)` 拼入该 assistant 消息内容。
   - 同一次构造内有 `expandedAttachmentIds` 去重，但去重粒度是最终收集到的附件 ID。

3. 流式最终回答完成时，最终 assistant 消息会挂上本轮工具附件。
   - 入口：`src/side-panel/state/appStoreStreaming.ts`
   - `complete` 分支调用 `finalizeAssistantMessage(...)` 时传入：
     - `toolAttachments: mergeToolAttachments(input.toolAttachments, message.toolAttachments)`
   - 这会让最终 assistant 消息保存 `toolAttachmentIds`，后续请求构造时会展开这些附件摘要。

4. 工具循环运行中估算和最终回答后估算的对象不同。
   - 工具循环运行中，`tool` 消息估算只看 `tool.content`，注释明确说明即时请求 payload 不发送 `toolAttachments`，避免重复估算。
   - 最终回答落库后，历史 assistant 消息会按持久化附件引用展开摘要。因此用户看到的运行中上下文和最终回答结束后的上下文可能突变。

5. 详情池不等于摘要不进上下文。
   - 入口：`src/shared/toolArtifacts.ts`
   - 当前摘要常量：
     - `TOOL_ATTACHMENT_CONTEXT_ITEM_LIMIT = 50`
     - `TOOL_ATTACHMENT_CONTEXT_TEXT_LIMIT = 500`
     - `TOOL_ATTACHMENT_CONTEXT_URL_LIMIT = 240`
   - `formatNetworkAttachmentForPromptSummary(...)`、`formatJsSourceAttachmentForPromptSummary(...)`、`formatSourceMapAttachmentForPromptSummary(...)` 会生成历史上下文摘要。
   - 截图中曾出现 `JS 源码片段 86`、`Network 请求详情 47`、自动化任务报告。即使完整详情不进入模型，50 项 * 500 字符的摘要也可能把上下文撑到几十 k 以上。

## 关键判断

1. “最终回答结束后理论上只增加 response token”只适用于纯聊天，不适用于当前工具附件链路。
   - 当前最终回答消息不仅有正文，还可能引用本轮全部工具附件。
   - 下一次请求上下文会展开这些附件摘要，所以增长量 = 最终回答正文 + 工具附件摘要 + 其他历史展开内容。

2. 附件池解决的是“大对象存储去重”和“完整详情不默认注入”，没有彻底解决“历史上下文摘要膨胀”。
   - Network/JS 详情池保住了完整 body/source 不长期进入上下文。
   - 但摘要本身仍然会进入上下文，而且当前摘要预算按附件分别限制，不按整次请求设置总预算。

3. 压缩边界可能被最终回答附件引用绕过。
   - 工具循环中途压缩后，`context_summary` 成为新的上下文边界。
   - 但最终回答消息落库在摘要之后，并可能引用整轮工具附件。
   - 下一次请求从最新摘要之后取历史时，可能通过最终回答消息重新展开压缩前产生的大附件摘要，等于压缩效果被最终落库抵消。

4. 可能存在摘要级重复，不一定是完整详情重复。
   - `buildChatRequestMessages` 用 `expandedAttachmentIds` 按附件 ID 去重。
   - 但工具过程消息和最终回答消息可能通过聚合产生不同的聚合附件 ID，即底层同一批原始附件可能被不同聚合 ID 表示。
   - 这会绕过当前去重 Set，导致同一批底层证据摘要重复进入上下文。

## 推荐修复方向

优先级从高到低：

1. 最终 assistant 消息的附件默认只用于 UI 展示，不参与后续模型上下文展开。
   - 推荐显式区分 UI 附件引用和上下文附件引用，例如：
     - `toolAttachmentIds`：用于 UI 展示、附件弹窗、导出。
     - `contextToolAttachmentIds`：允许进入后续模型上下文的少量引用。
   - 如果不想新增字段，也可以按消息类型策略处理：普通最终 assistant 消息上的附件引用默认不展开，只有工具过程消息或明确上下文引用才展开。

2. 压缩边界之后禁止旧附件摘要复活。
   - 构造最新 `context_summary` 之后的上下文时，如果某个附件来源早于摘要边界，默认不展开。
   - 最终回答消息即使引用压缩前附件，也只作为 UI 附件，不作为模型上下文附件。
   - 如果最终回答正文明确引用 `requestId/resourceId/contextId`，后续只保留短索引，不展开详情摘要。

3. 历史附件摘要设置“总预算”，而不是只设置单附件上限。
   - 当前是每个附件最多 50 项，每条片段最多 500 字符，多个附件叠加仍会很大。
   - 建议按一次 `buildChatRequestMessages` 设置工具附件摘要总预算，例如默认 3000-8000 tokens。
   - 超预算时按优先级裁剪：最终回答明确引用的 ID > 最近工具调用 > 接口类请求 > JS 命中 > 自动化报告 > 失败/噪音。

4. 附件去重改为基于底层原始附件 ID 或证据 ID，而不是只基于聚合后的附件 ID。
   - 需要避免 `aggregated-A-B` 与 `aggregated-A` / `aggregated-B` 被视为不同内容重复展开。
   - 可在聚合附件中保留 `sourceAttachmentIds`，或在构造请求时从 `toolCallRecords.attachmentIds`、原始附件 ID、requestId/resourceId 维度建立去重。

5. Token tooltip 增加来源拆分，方便以后定位暴涨。
   - 建议显示：
     - 系统提示
     - 页面上下文
     - 历史用户/助手正文
     - 最终回答正文
     - 工具过程正文
     - 工具附件摘要
     - 图片估算
     - Prompt 快照
     - 当前输入草稿
   - 这样用户看到 `133k/100k` 时能直接知道是谁撑爆。

## 建议的最小可落地方案

第一阶段只做低风险闭环：

1. 修改 `buildChatRequestMessages` 的附件展开策略：
   - 普通最终 assistant 消息上的 `toolAttachmentIds` 默认不展开为模型上下文。
   - `assistantMessageKind === "tool_call_turn"` 的附件仍可按摘要策略展开。
   - `assistantMessageKind === "context_summary"` 不展开附件。

2. 在最终回答消息落库时保留 UI 附件引用，但增加上下文展开排除规则。
   - 保持附件弹窗、导出、复制不受影响。
   - 只改变后续模型请求构造时的展开策略。

3. 给历史附件摘要加请求级总预算。
   - 先用简单预算：超过总预算后只保留附件标题、ID、数量、关键 requestId/resourceId 索引，不再保留片段正文。
   - 不需要立刻做复杂 AI 保留策略。

4. 补测试：
   - 最终回答消息引用 Network/JS 附件后，下一次 `buildChatRequestMessages` 不展开最终消息附件摘要。
   - 工具过程消息和最终回答消息引用同一附件时，下一次上下文只展开一次或按新策略不从最终消息展开。
   - 最新 `context_summary` 之后的最终回答消息引用压缩前附件时，不把压缩前附件摘要带回上下文。
   - Token 估算与 `buildChatRequestMessages` 生成的 request messages 保持一致。

## 注意事项

1. 不要把 UI 附件弹窗和模型上下文混在一起。
   - 附件仓库是给 UI、排障、导出和按需精查使用的。
   - 模型历史上下文只应拿到经过预算控制的轻量索引。

2. 不要再次改回“落库上下文估算”。
   - 用户已经明确要求底部上下文必须使用真实请求上下文/运行中上下文。
   - 修复方向应该是减少下一次真实请求会展开的内容，而不是隐藏或换口径。

3. 不要让压缩摘要失效。
   - `context_summary` 是上下文边界。
   - 边界之前的大工具附件不能通过边界之后的最终回答附件引用重新进入模型上下文。

4. 验证命令建议：
   - `npx vitest run tests/unit/shared/chatRequestMessages.test.ts`
   - `npx vitest run tests/unit/shared/chatRequestMessages.toolAttachments.test.ts`（若存在）
   - `npx vitest run tests/unit/side-panel/ChatComposerTokenUsage.test.tsx`
   - `npm run typecheck`
   - 涉及构建或 background/side-panel 串联时再跑 `npm run build:extension`

## 一句话结论

最终回答结束后上下文暴涨的根因不是“response token 增加太多”，而是“最终回答消息把本轮工具附件引用带入历史，下一次真实请求又展开了这些附件摘要；压缩边界还可能被最终回答附件引用绕过”。修复应围绕“UI 附件引用”和“模型上下文展开引用”分离，并给历史附件摘要设置总预算。
